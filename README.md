# ТЕХНИЧЕСКОЕ ЗАДАНИЕ
## на разработку программного продукта «Семейный менеджер»


## 1. ВВЕДЕНИЕ

### 1.1. Наименование программы
«Семейный менеджер» (Family Manager)

### 1.2. Область применения
Программа предназначена для использования семьями в целях централизованного управления семейными делами, включая:
- хранение и совместное использование файлов;
- ведение семейного бюджета и финансовую аналитику;
- организацию общения через семейный чат;
- управление устройствами «умного дома»;
- планирование событий через календарь;
- создание фото-презентаций по семейным событиям.

### 1.3. Краткая характеристика области применения
Программа ориентирована на семьи с детьми, нуждающиеся в единой цифровой платформе для координации повседневных дел, контроля расходов и сохранения семейной истории. Целевая аудитория включает три типа пользователей с различными уровнями доступа: дети, взрослые и администраторы.

## 2. ОСНОВАНИЕ ДЛЯ РАЗРАБОТКИ

### 2.1. Основание для проведения разработки
Разработка проводится в соответствии с учебным планом и заданием на курсовое проектирование.

### 2.2. Наименование темы разработки
«Разработка веб-приложения для управления семейными делами «Семейный менеджер»».

---

## 3. НАЗНАЧЕНИЕ РАЗРАБОТКИ

### 3.1. Функциональное назначение программы
Программа обеспечивает автоматизацию следующих функций:
- регистрация и аутентификация пользователей с разграничением прав доступа;
- загрузка, скачивание и управление файлами с тремя уровнями доступа (личный, семейный, публичный);
- учёт доходов и расходов, построение графиков и аналитика;
- обмен текстовыми сообщениями в реальном времени;
- управление устройствами «умного дома» (включение/выключение);
- планирование событий с возможностью прикрепления фотографий;
- генерация слайд-презентаций по выбранным событиям.

### 3.2. Эксплуатационное назначение программы
Программа эксплуатируется в домашних условиях под управлением современных веб-браузеров при наличии доступа к локальному серверу.

## 4. ТРЕБОВАНИЯ К ПРОГРАММЕ

### 4.1. Требования к функциональным характеристикам

#### 4.1.1. Управление пользователями
Программа должна обеспечивать:
- регистрацию новых пользователей с указанием имени, email и пароля;
- аутентификацию по email и паролю с выдачей JWT-токена;
- разграничение прав доступа на основе ролей (ребёнок, взрослый, администратор);
- привязку пользователя к семье по идентификатору семьи.

#### 4.1.2. Управление файлами
Программа должна обеспечивать:
- загрузку файлов размером до 20 МБ;
- скачивание файлов авторизованными пользователями;
- установку уровня доступа к файлу: «личный» (только владелец), «семейный» (все члены семьи), «публичный» (доступ по ссылке);
- генерацию публичной ссылки для файлов с открытым доступом;
- удаление файлов владельцем или администратором.

#### 4.1.3. Управление финансами
Программа должна обеспечивать:
- добавление доходов и расходов с указанием суммы, категории и описания;
- отображение общей суммы доходов, расходов и баланса;
- сравнение расходов с предыдущим месяцем;
- визуализацию структуры расходов в виде круговой диаграммы (Chart.js);
- хранение истории всех транзакций семьи.

#### 4.1.4. Семейный чат
Программа должна обеспечивать:
- обмен текстовыми сообщениями в режиме реального времени;
- отображение истории последних 100 сообщений при подключении;
- автоматическое переподключение при обрыве соединения WebSocket;
- отображение имени отправителя и времени отправки.

#### 4.1.5. Управление умным домом
Программа должна обеспечивать:
- добавление устройств с указанием типа (свет, термостат, замок);
- изменение статуса устройства (включено/выключено);
- удаление устройств (только администратором);
- хранение настроек устройств в формате JSON.

#### 4.1.6. Календарь событий
Программа должна обеспечивать:
- отображение календаря на текущий месяц с возможностью переключения;
- добавление событий с указанием названия, даты, времени и описания;
- отображение событий непосредственно в ячейках календаря;
- просмотр деталей события по клику;
- удаление событий (владельцем или администратором);
- отображение ближайших предстоящих событий.

#### 4.1.7. Календарь семейных событий с фотографиями
Программа должна обеспечивать:
- создание событий с названием, датой, местом и описанием;
- загрузку нескольких фотографий к каждому событию;
- выбор до 12 событий для создания презентации;
- ограничение до 3 фотографий на событие в презентации;
- генерацию HTML-презентации со слайдами информационных и фотографий;
- навигацию по слайдам (кнопки, клавиатура, свайпы).

#### 4.1.8. Ролевая модель доступа

| Роль | Права |
|------|-------|
| **Ребёнок (child)** | Просмотр семейных файлов, добавление транзакций (требуется подтверждение), отправка сообщений в чат, просмотр календаря, добавление событий |
| **Взрослый (adult)** | Все права ребёнка + загрузка файлов, удаление своих файлов, управление устройствами, редактирование событий |
| **Администратор (admin)** | Все права взрослого + удаление любых файлов, удаление любых устройств, удаление любых событий, управление пользователями |

### 4.2. Требования к надёжности

Программа должна:
- корректно обрабатывать ошибочные входные данные;
- не допускать потери данных при некорректном завершении работы;
- восстанавливать WebSocket-соединение при обрыве связи;
- обеспечивать контроль целостности данных на уровне внешних ключей БД.

### 4.3. Условия эксплуатации

#### 4.3.1. Клиентская часть
- Операционная система: Windows 10/11, macOS, Linux, iOS, Android;
- Браузер: Google Chrome (версия 90+), Mozilla Firefox (версия 88+), Safari (версия 14+);
- Доступ в интернет не требуется (работа в локальной сети).

#### 4.3.2. Серверная часть

**Сервер:**
- Процессор: 1.5 ГГц и выше;

**Клиент:**
- Устройство с доступом в локальную сеть;

- Бэкенд разрабатывается на языке Go (версия 1.21+);
- Фронтенд — на HTML5, CSS3, JavaScript ES6+;
- База данных — PostgreSQL 12+;
- Обмен данными осуществляется по протоколам HTTP/HTTPS и WebSocket;
- Программа не требует установки дополнительного ПО на клиенте;
- Формат обмена данными — JSON.

### 4.6. Требования к маркировке и упаковке
Не предъявляются (программа распространяется в виде исходных кодов).

### 4.7. Требования к транспортировке и хранению

## 3. ОПИСАНИЕ ЛОГИЧЕСКОЙ СТРУКТУРЫ

### 3.1. Модель базы данных

**Связи между таблицами:**

```
families (1) ─────< (N) users
    │                    │
    │                    │
    └──< (N) files       └──< (N) transactions
    └──< (N) messages    └──< (N) devices
    └──< (N) calendar_events
    └──< (N) family_events
                              │
                              └──< (N) event_photos
```

**Структура таблиц:**

```sql
-- Семьи
families (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    created_at TIMESTAMP
)

-- Пользователи
users (
    id SERIAL PRIMARY KEY,
    family_id INTEGER → families.id,
    username VARCHAR(50) UNIQUE,
    email VARCHAR(100) UNIQUE,
    password_hash VARCHAR(255),
    role VARCHAR(20), -- 'child', 'adult', 'admin'
    created_at TIMESTAMP
)

-- Файлы
files (
    id SERIAL PRIMARY KEY,
    family_id INTEGER → families.id,
    user_id INTEGER → users.id,
    name VARCHAR(255),
    path VARCHAR(500),
    access_type VARCHAR(20), -- 'private', 'family', 'public'
    share_token VARCHAR(100) UNIQUE,
    created_at TIMESTAMP
)

-- Транзакции
transactions (
    id SERIAL PRIMARY KEY,
    family_id INTEGER → families.id,
    user_id INTEGER → users.id,
    amount DECIMAL(10,2),
    type VARCHAR(20), -- 'income', 'expense'
    category VARCHAR(50),
    description TEXT,
    date DATE,
    created_at TIMESTAMP
)

-- Сообщения чата
messages (
    id SERIAL PRIMARY KEY,
    family_id INTEGER → families.id,
    user_id INTEGER → users.id,
    username VARCHAR(50),
    message TEXT,
    created_at TIMESTAMP
)

-- Устройства умного дома
devices (
    id SERIAL PRIMARY KEY,
    family_id INTEGER → families.id,
    name VARCHAR(100),
    type VARCHAR(50), -- 'light', 'thermostat', 'lock'
    status VARCHAR(20), -- 'on', 'off'
    settings JSONB,
    created_at TIMESTAMP
)

-- События календаря
calendar_events (
    id SERIAL PRIMARY KEY,
    family_id INTEGER → families.id,
    user_id INTEGER → users.id,
    title VARCHAR(200),
    description TEXT,
    event_date DATE,
    event_time TIME,
    created_at TIMESTAMP
)

-- Архивные семейные события
family_events (
    id SERIAL PRIMARY KEY,
    family_id INTEGER → families.id,
    user_id INTEGER → users.id,
    title VARCHAR(200),
    description TEXT,
    event_date DATE,
    location VARCHAR(200),
    created_at TIMESTAMP
)

-- Фотографии к событиям
event_photos (
    id SERIAL PRIMARY KEY,
    event_id INTEGER → family_events.id,
    user_id INTEGER → users.id,
    photo_path VARCHAR(500),
    photo_url VARCHAR(500),
    caption VARCHAR(500),
    sort_order INTEGER,
    created_at TIMESTAMP
)
```
